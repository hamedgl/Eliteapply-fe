  ==================================================
  1. OBJECTIVE
  ==================================================

  Create a secure operational console that lets authorized administrators:

  - Manage user activation and administrator access.
  - Inspect and override user entitlements.
  - Inspect AI usage.
  - Reset a user’s current usage window.
  - Monitor, cancel, and retry asynchronous operations.
  - Manage feature flags and emergency kill switches.
  - Moderate and merge catalogue records.
  - Inspect redacted reference event history.
  - Redrive allowlisted dead-letter queues.
  - Trigger due-reminder dispatch.
  - Manage launch gates and inspect launch readiness.
  - Review the immutable administrative audit trail.

  This is an operations console, not a private-content browser.

  Never expose or invent access to:

  - Essays, prompts, raw opportunity import text, transcripts, confidential references, or private document content.
  - Raw SQS/DLQ message bodies.
  - Authentication tokens, storage keys, signed URLs, or provider secrets.
  - Arbitrary database CRUD.
  - User hard deletion; no admin hard-delete API exists.
  - Bulk destructive operations; no bulk API exists.

  ==================================================
  2. AUTHENTICATION AND ACCESS CONTROL
  ==================================================

  API base path:

    /api/v1

  All admin endpoints require:

    Authorization: Bearer <access_token>

  Reuse the existing frontend authentication/session implementation. Do not store a second token or create an admin-specific login flow.

  On admin entry:

  1. Fetch GET /api/v1/users/me using the existing authenticated-user query.
  2. Require `is_admin === true`.
  3. If not authenticated, use the existing login/refresh flow.
  4. If authenticated but not an admin, render a proper 403 page and remove admin navigation.
  5. Client-side guards are UX only. The backend remains authoritative.
  6. Handle a backend 403 even when the cached user says they are an admin; permissions may have changed.
  7. Clear all admin query caches on logout or identity change.
  8. Never persist user, usage, entitlement, audit, or operational data to localStorage.

  Every request should send a generated UUID in:

    X-Correlation-ID: <uuid>

  Capture the response `X-Correlation-ID`. Show it as a copyable support reference on error screens and mutation failures.

  Retry policy:

  - GET requests: up to two retries for network errors and transient 5xx responses, with backoff.
  - Never automatically retry POST, PUT, or PATCH admin mutations.
  - Never retry 401 or 403 automatically beyond the existing single token-refresh attempt.
  - Disable mutation controls while a request is in flight to prevent double submission.

  4. VISUAL AND INTERACTION DIRECTION
  ==================================================

  Design a calm, serious operations console:

  - Dense enough for operational work, but not cramped.
  - Neutral surfaces with restrained borders.
  - Red only for dangerous or failed states.
  - Amber for warnings, rollout risk, and waived gates.
  - Green for confirmed success and passed gates.
  - Use text plus icons for status; never rely on color alone.
  - Use tabular numerals for tokens, costs, percentages, and timestamps.
  - Show timestamps in the administrator’s locale and timezone, with exact UTC on hover.
  - Use proper skeletons rather than layout-shifting spinners.
  - Never render JSON or metadata with `dangerouslySetInnerHTML`.
  - Render unknown metadata using a safe JSON tree or escaped monospace block.
  - Provide clear focus states, labels, descriptions, and screen-reader announcements.
  - All dialogs must trap focus and restore it when closed.
  - Respect reduced-motion settings.

  Do not turn the interface into a collection of decorative cards. Prefer tables, split panes, compact summaries, timelines, and focused action panels.

  6. CURSOR PAGINATION
  ==================================================

  All cursor-paginated responses use:

  {
    "items": [],
    "next_cursor": "opaque-value-or-null",
    "has_more": false,
    "total": null
  }

  Rules:

  - Treat cursors as opaque.
  - Never construct, decode, or modify cursors.
  - Do not display fake page counts when `total` is null.
  - “Next” uses `next_cursor`.
  - Support “Previous” by keeping a client-side stack of previously used cursors.
  - Reset the cursor stack when filters change.
  - Keep filters and the active cursor in URL search parameters where practical.
  - Page sizes must stay within API limits.
  - Do not fetch every page to calculate a total.
  - Use `has_more`, not the item count, to determine whether “Next” is enabled.

  Rules:

  - Do not send both values.
  - Email is exact-match support lookup, not fuzzy search.
  - Validate UUID syntax before requesting.
  - On success, navigate to `/admin/users/:id`.
  - A missing user returns 404.

  User detail:

  GET /api/v1/admin/users/{user_id}

  Response example:

  {
    "id": "uuid",
    "email": "student@example.com",
    "full_name": "Student Name",
    "is_active": true,
    "is_admin": false,
    "created_at": "2026-07-18T12:00:00Z",
    "last_login_at": "2026-07-18T11:00:00Z"

  1. Account
  2. Entitlement
  3. AI usage
  4. Audit history

  Account actions:

  PATCH /api/v1/admin/users/{user_id}

  Request:

  {
    "is_active": false,
    "reason": "Support-requested account suspension"
  }

  or:

  {
    "is_admin": true,
    "reason": "Approved administrator access"
  }

  or both fields together.

  Rules:

  - `reason` is required, 3–1000 characters.
  - Send only changed boolean fields.
  - Never send `is_active: null` or `is_admin: null`.
  - At least one of the two state fields must be present.
  - Use the server response as authoritative.
  - Deactivation revokes the account’s usable session state.
  - The backend blocks an administrator from deactivating or demoting their own account.
  - Compare the target ID with GET `/api/v1/users/me` and disable self-demotion/self-deactivation before submission.
  - Still handle server 422 because the backend is authoritative.

  Confirmation levels:

  - Activate: confirmation dialog with reason.
  - Deactivate: danger dialog; show email and impact; require typed email confirmation.
  - Grant admin: high-risk dialog; require typed email and reason.
  - Revoke admin: high-risk dialog; require typed email and reason.
  - Never use a one-click toggle for role or active-state changes.
  - Do not optimistically update account or role state.

  After success:

  - Replace cached user detail with the response.
  - Invalidate user-list queries.
  - Invalidate audit queries.
  - Announce success accessibly.

  Response example:

  {
    "id": "uuid",
    "type": "writing_generation",
    "status": "failed",
    "user_id": "uuid",
    "created_at": "2026-07-18T12:00:00Z",
    "completed_at": "2026-07-18T12:01:00Z",
    "failure_reason": "Safe failure explanation"
  }

  Never expect or display request payloads, prompts, raw source text, essays, or transcripts.

  Polling:

  - Poll visible active operations only.
  - Suggested active states include queued, processing, ready, and in_progress.
  - Poll every 5 seconds while the tab is visible.
  - Back off to 15–30 seconds after repeated unchanged results.
  - Stop polling terminal operations.
  - Pause polling when the browser tab is hidden.
  - Do not create a WebSocket dependency; no admin realtime API exists.

  Retry:

  POST /api/v1/admin/operations/{operation_id}/retry

  No request body.

  Rules:

  - Retry is valid only for failed `writing_generation` and failed `opportunity_import`.
  - Interview operations never support retry.
  - The response is authoritative.
  - A writing-generation retry can return a new operation ID.
  - Navigate to or render the returned operation, not the old ID.
  - Invalid state may return 409 or 422.
  - Require confirmation, but the API does not accept a custom reason for this action.

  Cancel:

  POST /api/v1/admin/operations/{operation_id}/cancel

  No request body.

  Rules:

  - Cancellation is best effort.
  - Hide or disable cancellation for clearly terminal states.
  - Still handle a terminal no-op response from the backend.
  - Require confirmation.
  - Do not optimistically mark an operation cancelled.
  - Invalidate operation list, detail, and audit queries on success.

  ==================================================
  17. API ERROR HANDLING
  ==================================================

  Application errors generally use:

  {
    "error": "Conflict",
    "detail": "Human-readable explanation",
    "correlation_id": "uuid",
    "code": "optional_machine_code",
    "details": {}
  }

  FastAPI request-validation failures may instead use:

  {
    "detail": [
      {
        "loc": ["body", "field"],
        "msg": "Validation message",
        "type": "validation_type"
      }
    ]
  }

  The API client must normalize both formats into one frontend error model.

  Handle:

  - 400: malformed request
  - 401: session invalid or expired
  - 403: authenticated but no longer authorized
  - 404: target not found
  - 409: state conflict, such as retrying a non-failed operation or merging into a non-canonical target
  - 422: invalid fields, missing reason, invalid cursor, invalid UUID, self-demotion, or unsupported action
  - 500: unexpected server failure
  - 502/503: dependency unavailable

  Behavior:

  - Show actionable server `detail`.
  - Do not expose stack traces or raw response bodies.
  - Display and allow copying `correlation_id`.
  - Map field-level 422 errors to the relevant form controls.
  - Keep dialogs open after validation errors.
  - For 409, retain user input and offer refetch/review.
  - For 403, clear admin query data and move to the forbidden page.
  - For 404 on a detail route, show a proper not-found view.
  - For network failures, distinguish “request may not have reached the server” from a confirmed server rejection.
  - Never tell the user a mutation succeeded until the server confirms it.
